## Undersøg: GitLab Continuous Integration & Deployment
### Hvad betyder CI og hvordan kan man bruge versionsstyrring til dette?
Continuous Integration er en praksis hvori hver ændring i et program automatisk bliver bygget og testet continuerligt. Det vil sige at hver ændring bliver testet for om de stemmer overens med de retningslinjer der på forhånd er blevet etableret, så ny kode altid bliver holdt op imod en hvis standard. <br>
Dette bruges til at sikre et godt workflow da sandlynligheden for at lave fejl/bugs reduceres og bibeholder kodens integritet efter at have lavet et push til repo, da man kan lave scripts til at teste koden automatisk.
## Undersøg GitHub / Gitlab
### Hvad er forskellene?
#### GitHub:
<ul>
  <li>Hvis man bruger GitHub skal man selv vælge til CI/CD værktøj </li>
  <li>Fokus på workflow hastighed</li>
</ul>

#### GitLab:
<ul>
  <li>GitLab er CI/CD indbygget, samt DevOps. </li>
  <li>Fokus på workflow pålidelighed</li>
</ul>

## Er der fordele og ulemper ved at bruge den ene frem for den anden?
Der er umidelbart meget lille forskel mellem GitHub og GitLab, så forskelle og ulemper er små. I GitLab er interfacet lidt langsommere. I GitHub har der engang været en begrænsning for hvor mange deltagere(contributors), der kan være på et privat repo (det er der ikke længere, lige som i GitLab). GitHub har mest fokus på at man laver en masse commit, medens GitLab skulle have mere fokus på mange branches (men man kan begge dele på begge to, så ultimativt er der tæt på ingen forskel)
